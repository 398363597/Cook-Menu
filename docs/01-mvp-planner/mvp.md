# Cook_Menu 第一版 MVP

## 1. 产品定位

Cook_Menu 是一个个人自用菜谱记录 App，核心目标是：把“我做过、想复做、下次要改进”的菜谱稳定记录下来。

它不是美食社区，也不是内容平台。第一版只服务一个人：方便自己快速找到菜、照着做、记录失败经验，并在下次做同一道菜时少踩坑。

## 2. 用户场景

- 想记录一道新菜：做完一道菜后，把成品图、食材、调料、步骤、失败点记录下来。
- 想复做一道菜：打开 App，按分类或关键词找到菜谱，查看用料和步骤。
- 想改进一道菜：查看上次失败经验，例如“太咸”“火太大”“鸡胸肉太柴”，下次做时提前注意。
- 想快速决定吃什么：通过早餐、家常菜、快手菜、汤粥、面食等分类筛选已有菜谱。
- 想保留自己的做法：记录适合自己口味的版本，而不是收藏网上一堆不一定会做的菜谱。

## 3. 第一版必须功能

### 菜谱列表

- 展示已有菜谱。
- 每个菜谱至少显示：菜名、图片、分类、简短备注或最近改进点。
- 支持按分类筛选。

### 菜谱详情

- 展示完整菜谱信息。
- 必须包含：
  - 菜谱图片
  - 菜名
  - 分类
  - 食材列表
  - 调料列表
  - 做菜步骤
  - 失败经验 / 下次改进
  - 可选备注

### 新增菜谱

- 可以创建一条完整菜谱。
- 食材和调料必须分开填写。
- 步骤支持多条记录。
- 失败经验 / 下次改进可以为空，但字段必须存在。

### 编辑菜谱

- 可以修改已有菜谱。
- 重点支持修改步骤、用料、失败经验和下次改进。

### 删除菜谱

- 可以删除不再需要的菜谱。
- 第一版可以用简单确认，避免误删。

### 分类筛选

- 预置基础分类：
  - 早餐
  - 家常菜
  - 快手菜
  - 汤粥
  - 面食
  - 其他
- 一道菜第一版可以先只属于一个分类，降低复杂度。

### 本地数据保存

- 数据保存在本地。
- 页面刷新后数据不能丢。
- 第一版优先稳定，不要求账号和云同步。

## 4. 第一版不做功能

- 不做社区、点赞、评论、关注、粉丝、厨友动态。
- 不做账号系统。
- 不做多人使用。
- 不做复杂推荐算法。
- 不做菜谱公开发布。
- 不做营养计算。
- 不做购物清单。
- 不做库存管理。
- 不做复杂标签体系。
- 不做评分体系。
- 不做视频菜谱。
- 不做外部菜谱抓取。
- 不做多设备同步。
- 不做商业化、运营位、广告、热门榜单。

## 5. 页面结构

### 菜谱列表页

- 解决“我有哪些菜谱”和“今天想找哪类菜”的问题。
- 包含分类筛选、菜谱卡片、新增入口。

### 菜谱详情页

- 解决“照着这道菜复做”的问题。
- 清晰展示图片、食材、调料、步骤、失败经验 / 下次改进。

### 新增 / 编辑菜谱页

- 解决“记录或修正一道菜”的问题。
- 表单结构围绕菜谱核心字段，不做复杂配置。

### 分类筛选区

- 可以作为列表页的一部分，不一定单独成页。
- 解决“快速缩小范围”的问题。

## 6. 核心使用流程

1. 用户打开 Cook_Menu，进入菜谱列表页。
2. 用户点击“新增菜谱”。
3. 用户填写菜名、上传或选择菜谱图片、选择分类。
4. 用户分别填写食材和调料。
5. 用户填写做菜步骤。
6. 用户填写失败经验 / 下次改进。
7. 用户保存菜谱。
8. 下次做饭时，用户通过分类筛选或列表找到菜谱。
9. 用户进入详情页，按照食材、调料和步骤复做。
10. 做完后，如果发现问题，编辑“失败经验 / 下次改进”。

最小闭环是：新增菜谱 -> 本地保存 -> 分类找到 -> 查看详情复做 -> 编辑改进记录。

## 7. 第二版候选功能

- 搜索菜谱名称。
- 多分类或标签，例如“鸡肉”“少油”“空气炸锅”。
- 收藏 / 常做标记。
- 最近做过记录。
- 做菜次数统计。
- 每次复做日志，记录不同日期的调整。
- 图片支持多张，例如成品图、过程图。
- 食材用量单位规范化。
- 步骤拖拽排序。
- 数据导入 / 导出。
- 本地备份文件。
- 多设备同步，若后续真的需要，再考虑账号或云存储。
- 简单“今天吃什么”随机选择，但只基于自己的菜谱库。

## 8. 给数据模型 Agent 的交接说明

第一版数据模型请围绕“个人复做菜谱”设计，避免社区化字段。

核心实体建议为 `Recipe`，至少包含：

- `id`：菜谱唯一标识
- `name`：菜名
- `image`：菜谱图片地址或本地存储引用
- `category`：分类
- `ingredients`：食材列表
- `seasonings`：调料列表
- `steps`：步骤列表
- `failureNotes`：失败经验
- `nextImprovements`：下次改进
- `notes`：其他备注，可选
- `createdAt`：创建时间
- `updatedAt`：更新时间

注意事项：

- 食材和调料必须是两个独立字段，不要混在一个用料列表里。
- 步骤要支持有序列表。
- 失败经验和下次改进可以分成两个字段，方便详情页突出展示。
- 分类第一版先用单选字符串即可，不需要复杂标签关系。
- 第一版优先本地保存和数据不丢，不需要账号、用户表、社交关系表。
