## 3. 问题清单

1. 高优先级：页面仍出现社区化文案。
   - 文件位置：`D:\CodeX\Cook-Menu\index.html:1538`
   - 现象：设置页“个人版范围”中出现“关注、粉丝、点赞、评论、社区动态、发布到社区”等词。
   - 复现：打开 App -> 点击底部“我的” -> 查看“个人版范围”。
   - 判断：虽然语义是“不包含”，但验收要求检查是否还存在社区化功能或文案；当前属于文案存在。

2. 中优先级：调料为空会阻止保存，可能不符合真实个人菜谱场景。
   - 文件位置：`D:\CodeX\Cook-Menu\index.html:1334`
   - 复现：新增菜谱 -> 填菜名、食材、步骤 -> 清空调料 -> 点击保存。
   - 实际：弹窗“请至少填写一条调料。”
   - 风险：米饭、水煮蛋、蒸红薯等菜谱可能没有独立调料，用户会被迫填写“无”或“适量”，数据不够干净。
   - 备注：MVP 强调食材和调料分开，但未明确调料必须至少一条；建议产品口径确认。

3. 低优先级：首次打开默认写入样例菜谱，可能污染个人数据。
   - 文件位置：`D:\CodeX\Cook-Menu\index.html:802`、`D:\CodeX\Cook-Menu\index.html:805`
   - 现象：清空存储后首次打开会自动保存“番茄鸡蛋面”样例。
   - 风险：个人自用 App 容易把样例当成真实数据参与搜索、筛选、统计。

4. 低优先级：空菜名提示依赖浏览器原生校验，提示样式不统一。
   - 文件位置：`D:\CodeX\Cook-Menu\index.html:1225`
   - 现象：空菜名不会进入 JS 弹窗，而是触发浏览器原生 required 提示。
   - 风险：功能上可接受，但和食材、调料、步骤的弹窗提示体验不一致。

## 4. 高优先级修复

- 移除设置页中具体社区词汇。建议将 `D:\CodeX\Cook-Menu\index.html:1538` 改成更中性的个人版说明，例如“第一版只做自己的菜谱记录、本地保存、分类检索和复做提醒，不包含互动或公开发布功能。”
- 确认“调料是否允许为空”。如果允许，删除 `D:\CodeX\Cook-Menu\index.html:1333-1335` 的阻止保存逻辑，并在详情页保留“还没有记录调料。”空态。

## 5. 中低优先级优化

- 首次启动建议改为空列表引导，不自动写入样例菜谱；如需示例，可做成可删除的演示入口或静态空态说明。
- 空菜名校验建议改为与其他字段一致的应用内提示，避免不同浏览器提示样式不一致。
- 图片目前用 `localStorage` base64 保存，MVP 可用；如果日常使用菜谱和图片变多，建议迁移到 IndexedDB，降低容量风险。
